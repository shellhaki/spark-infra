SparkCell Project Plan: Database Isolation Engine
Phase 1: Engine Foundation & IPC Loop
Milestone 1: Client-Daemon IPC & State Tracking
Objective: Build the command-line client, the persistent background daemon, and the local state machine database so they can securely communicate over a Unix Domain Socket.

Task 1.1: Build the user-facing CLI command structure using a library like spf13/cobra. Supported user inputs:

sparkcell run postgres --port <port> --memory <limit> --cpu <limit>

sparkcell list

sparkcell stop <cell_id>

Task 1.2: Initialize a persistent background daemon architecture that runs as a root system service and listens exclusively on a Unix domain socket at /var/run/sparkcell.sock.

Task 1.3: Embed an internal SQLite instance inside the daemon to track state tracking fields: id (TEXT), host_pid (INTEGER), port (INTEGER), status (TEXT), memory_limit (TEXT), cpu_limit (TEXT), and created_at (DATETIME).

Task 1.4: Standardize the message contract (JSON over Unix socket). The client serializes flags into JSON; the daemon processes the request, commits an entry to SQLite, and replies with a success frame.

Milestone 2: Automated Storage Provisioning
Objective: Replace hardcoded paths with a clean, dynamic filesystem manager that clones the baseline OS image into distinct multi-tenant cell roots.

Task 2.1: Establish a dedicated platform root directory structure on the host machine at /var/lib/sparkcell/. Create two subdirectories: /var/lib/sparkcell/images/ (holds base_linux_environment.gz) and /var/lib/sparkcell/cells/.

Task 2.2: Write a storage utility package inside the daemon that intercepts a fresh cell request, generates a unique time-stamped string hash (cell_20260614_xxxx), creates a target folder under /var/lib/sparkcell/cells/<cell_id>, and extracts the base rootfs archive into it using an automated tar unpacking pipeline.

Task 2.3: Implement a deletion pipeline. When sparkcell stop <cell_id> is issued, the daemon kills the process, cleanly unmounts system paths, and purges the directory to reclaim storage space.

Phase 2: Runtime Isolation & Resource Control
Milestone 3: Re-entrant Process Isolation Engine
Objective: Integrate your custom low-level namespace splitting and jail-anchoring logic into the daemon orchestration loop.

Task 3.1: Implement the re-entrant fork pattern via /proc/self/exe child <cell_path> inside the backend daemon code.

Task 3.2: Wire up the namespace configuration array passing the complete structural isolation set: CLONE_NEWUTS (hostname), CLONE_NEWPID (process tree), CLONE_NEWNS (mount namespace), and CLONE_NEWNET (network loop isolation).

Task 3.3: Standardize the internal system anchoring array inside the child context execution: call Chroot to lock the filesystem path, Chdir("/") to drop anchor inside the new root, bind-mount the host's Postgres directories (/usr/lib/postgresql), and mount a fresh virtual /proc instance.

Milestone 4: Linux cgroups v2 Resource Throttling
Objective: Hard-cap memory allocations and CPU scheduling slices inside the cell using the kernel cgroup interface to enforce strict hardware multi-tenancy.

Task 4.1: Build a cgroups controller package inside the daemon that maps to the modern unified path: /sys/fs/cgroup/sparkcell/.

Task 4.2: When a cell is created, create a dedicated leaf folder matching the identifier: /sys/fs/cgroup/sparkcell/<cell_id>/.

Task 4.3: Enforce memory constraints by converting the user's --memory string flag (e.g., 512M) into raw bytes and writing that value directly into the cgroup parameter file:

Write bytes value to /sys/fs/cgroup/sparkcell/<cell_id>/memory.max

Write an optimized swap restriction value to memory.swap.max (set to 0 to prevent database performance degradation due to disk swapping).

Task 4.4: Enforce CPU processing slice constraints by translating the user's --cpu parameter (e.g., 0.5 cores) into standard microsecond period weights and writing them to the cgroup parameters:

Write the calculated period tracking values to /sys/fs/cgroup/sparkcell/<cell_id>/cpu.max (e.g., 50000 100000 to constrain execution to 50% of a single core's scheduling cycle).

Task 4.5: Anchor the process by writing the cell's newly captured Host PID into the cgroup process registry file: /sys/fs/cgroup/sparkcell/<cell_id>/cgroup.procs immediately before allowing the payload to boot.

Phase 3: Network Interconnect & Automation
Milestone 5: Virtual Networking Mesh & NAT Gateway
Objective: Bridge the cell's network namespace black hole back to the host machine via software ethernet cables, virtual switches, and packet routing rules.

Task 5.1: Write a network orchestration package using native Go netlink frameworks or clean execution commands to instantiate a virtual network switch bridge interface (sparkbr0) assigned to a standard local gateway subnet (e.g., 10.200.0.1/24).

Task 5.2: Programmatically generate unique software patch cables (veth pairs) for each runtime request: v_host_<id> and v_cell_<id>.

Task 5.3: Attach the host interface end (v_host_<id>) to the sparkbr0 virtual switch. Push the alternative end (v_cell_<id>) directly into the cell's isolated network namespace using its tracked Host PID.

Task 5.4: Enter the cell's private network stack space using a nsenter wrapper execution: bring up the local loopback interface (lo), rename the virtual wire to standard eth0, attach a unique static IP address from the subnet pool (e.g., 10.200.0.2/24), and configure the default routing table gateway to point straight back to 10.200.0.1.

Task 5.5: Automate host machine packet forwarding translation rules via iptables. Dynamically append a Destination Network Address Translation (DNAT) mapping rule to pass incoming traffic hitting your host at the user's requested external port flag straight down the bridge to 10.200.0.X:5432.

Milestone 6: End-to-End PostgreSQL Automated Lifecycle
Objective: Wire up the final system layer to dynamically initialize, configure, and boot the bare-metal database cluster inside the fully walled cell.

Task 6.1: Automate database cluster generation within the filesystem provisioning layer. Have the daemon trigger the bind-mounted initdb command targeting an internal directory structure (e.g., /var/lib/postgresql/data) inside the cell jail folder before starting the namespace execution.

Task 6.2: Set up the daemon to dynamically edit the generated configuration file (/var/lib/postgresql/data/postgresql.conf) inside the cell root to allow binding to all addresses (listen_addresses = '*') so it accepts the incoming virtual bridge connection cleanly.

Task 6.3: Swap the child entry command argument from a raw debugging shell (/bin/bash) to the execution path of the Postgres database process itself, running under the isolated unprivileged system user context (su - postgres equivalents).

Task 6.4: Run a validation suite: issue sparkcell run postgres --port 8001 --memory 512M --cpu 0.5, ensure the daemon outputs a clean tracking tracking state, verify the database process is constrained to 512MB inside the cgroup runtime logs, and connect to it from an external host database shell using psql -h 127.0.0.1 -p 8001 -U postgres.
